# Codia 权限系统 Spec

## 背景

Codia 是一个终端 AI 编程助手，模型可以通过工具调用来读写文件、执行 shell 命令。当前工具执行链（AgentLoop → ToolScheduler → executeTool → tool.execute()）没有权限检查层，模型可以无限制地执行任意操作。这在 AI 编程助手的场景下存在风险——模型可能误操作删除文件、执行危险命令、或在项目目录外读写文件。

## 目标

为 Codia 构建一套五层防御的权限系统，在不终止 Agent Loop 的前提下拦截或确认高风险操作：

1. **不可绕过的黑名单**——硬性拦截已知高危命令模式
2. **路径沙箱**——限制文件操作在项目目录内，防符号链接逃逸
3. **多档权限模式**——提供 default / acceptsEdit / plan / bypassPermissions 四档
4. **可配置规则引擎**——支持用户级、项目级、本地覆盖三层 YAML 规则
5. **人在回路**——规则未明确命中时，让用户决定放行还是拒绝

权限被拒绝时，模型收到错误反馈并有机会调整策略，不终止 Agent Loop。

## 功能需求

### F1: 危险命令黑名单

用正则表达式在命令执行前硬性拦截已知高危模式，例如 `rm -rf /`、`chmod 777 /`、`mkfs` 等。这一层的拦截不可被任何配置放开，任何权限模式下都生效。仅作用于 shell 命令类工具（Bash/RunCommand）。

### F2: 路径沙箱

限制文件读写操作只能落在项目目录内。输入路径先解析符号链接，再用前缀比对确保真实路径在项目目录下。适用于所有文件类工具（read_file、write_file、edit_file、glob、grep 等）。超出沙箱的操作拒绝执行。

若项目目录本身是符号链接，以解析后的真实路径为沙箱边界。

### F3: 可配置规则引擎

规则描述「什么工具 + 什么参数/路径 + 放行还是拒绝」。规则格式为 `工具名(模式)`，例如 `Bash(git *)` 表示匹配所有 git 命令。每条规则的结果只有 `allow` 或 `deny`。模式支持：

- 精确匹配：`Bash(git status)`
- glob 匹配：`Bash(git *)`、`Read(**/*.ts)`
- 只写工具名不写模式时匹配该工具的所有调用：`Bash` 即 `Bash(*)`

规则按三层 YAML 文件存放：

| 层 | 路径 | 说明 |
|----|------|------|
| 本地覆盖（最高） | `{project}/.codia/permissions.local.yaml` | 个人本地偏好，不提交 git |
| 项目级 | `{project}/.codia/permissions.yaml` | 团队共享，提交 git |
| 用户全局（最低） | `~/.codia/permissions.yaml` | 跨项目的个人默认偏好 |

**否决规则：** 任意一层 deny，即使其他层有 allow 也覆盖不掉。只有三层都没有 deny 时，allow 才生效。

### F4: 多档权限模式

提供四种权限模式，影响「规则未命中时」的默认行为：

| 模式 | 只读工具 | 编辑工具 | Bash 命令 | 危险命令 |
|------|---------|---------|----------|---------|
| default | 放行 | 入回路 | 入回路 | 拦截 |
| acceptsEdit | 放行 | 放行 | 入回路 | 拦截 |
| plan | 放行 | 拦截 | 拦截 | 拦截 |
| bypassPermissions | 放行 | 放行 | 放行 | 拦截 |

- 黑名单（F1）在所有模式下生效
- 规则（F3）的判定优先于模式默认行为
- 模式可通过 CLI 参数（`--permission-mode`）或运行时命令切换

### F5: 人在回路

当规则引擎和权限模式都无法明确判定时，暂停执行让用户做决定。用户选择：

- **是（Yes）**——本次放行，不创建规则，仅针对当前这次调用
- **否（No）**——本次拒绝，返回权限错误给模型让其调整策略
- **始终允许（Always Allow）**——创建一条 allow 规则写入 `permissions.local.yaml`，此后同类调用自动放行

用户拒绝时，工具返回「权限被拒绝」的结构化错误，模型可以据此调整策略。

### F6: 五层决策链

权限检查按以下顺序执行，上一层能决策就立即返回结果，不能决策才交给下一层：

```
Layer 1: 危险命令黑名单 (F1)
  ├─ 命中 → 立即 deny（不可绕过）
  └─ 未命中 → 进入 Layer 2

Layer 2: 路径沙箱 (F2)
  ├─ 路径逃逸 → 立即 deny
  └─ 安全 → 进入 Layer 3

Layer 3: 可配置规则引擎 (F3)
  ├─ 任意层 deny 命中 → 立即 deny
  ├─ 任意层 allow 命中（且无 deny）→ 立即 allow
  └─ 无匹配 → 进入 Layer 4

Layer 4: 权限模式默认行为 (F4)
  ├─ 模式说放行 → allow
  ├─ 模式说拒绝 → deny
  └─ 模式说需确认 → 进入 Layer 5

Layer 5: 人在回路 (F5)
  ├─ 用户选是 → allow（本次）
  ├─ 用户选否 → deny
  └─ 用户选始终允许 → allow + 写入 permissions.local.yaml
```

每层返回结果只有 `allow` 或 `deny`，无中间状态。

### F7: 拒绝不终止 Loop

权限被拒绝时，不终止 Agent Loop。工具返回结构化错误结果给模型，模型可以选择：

- 用另一种方式实现相同目标
- 用另一个工具替代
- 向用户解释为什么需要该权限

## 非功能需求

### N1: 性能

权限检查在工具调用路径上同步执行，必须在毫秒级完成。对于百条规则以内的规模，单次检查应 < 1ms。

### N2: 可测试性

权限检查逻辑不耦合 I/O。规则匹配、优先级判定、黑名单检测均为纯函数，可单独单元测试。YAML 文件读取有独立可 mock 的边界。

### N3: 可扩展性

新增一个权限模式只需增加配置项，不修改决策链流程。新增工具类型时，权限模式的行为定义可扩展。

### N4: 错误提示

权限被拒绝时返回给模型的信息必须清晰——说明是第几层拦截、原因、以及模型可以尝试什么替代方案。

## 不做的事

- **网络请求限制**——不拦截或不审查 HTTP/API 调用，留给后续
- **资源配额**——不限制 CPU、内存、磁盘使用量，留给后续
- **审计日志**——不记录权限决策历史和用户选择，留给后续
- **权限模式定时切换**——不支持按时间段自动切换模式
- **规则热更新**——配置文件变更后需重启会话才生效，不做文件监听
- **远程规则源**——只读本地 YAML 文件，不从远程 URL 拉取规则

## 验收标准

### 对应 F1（黑名单）
- AC1.1: 执行 `rm -rf /` 被 Layer 1 拦截，无论当前权限模式是什么
- AC1.2: 正常命令 `git status` 不被黑名单误拦

### 对应 F2（路径沙箱）
- AC2.1: write_file 目标路径 `/etc/passwd`（绝对路径在项目外）被 Layer 2 拒绝
- AC2.2: write_file 目标 `../outside/file.txt`（相对路径逃逸）被 Layer 2 拒绝
- AC2.3: 通过符号链接指向项目外目录的文件写入被拒绝
- AC2.4: 正常文件操作在项目目录内放行

### 对应 F3（规则引擎）
- AC3.1: `permissions.yaml` 中 `Bash(git *): allow` 对 `git status` 生效，返回 allow
- AC3.2: 本地覆盖层的 deny 覆盖项目级 allow
- AC3.3: glob 模式 `Read(**/*.md)` 匹配 `read_file docs/spec.md`
- AC3.4: 无匹配规则时进入下一层决策

### 对应 F4（权限模式）
- AC4.1: default 模式下，只读工具直接放行，Bash 命令需确认
- AC4.2: acceptsEdit 模式下，编辑工具也直接放行
- AC4.3: plan 模式下，编辑和 Bash 均被拒绝
- AC4.4: bypassPermissions 模式下，仅危险命令被拦截，其余放行

### 对应 F5（人在回路）
- AC5.1: 需确认时暂停执行，用户选「是」则本次放行
- AC5.2: 用户选「否」则返回权限错误，Loop 继续
- AC5.3: 用户选「始终允许」则规则写入 permissions.local.yaml

### 对应 F6（决策链）
- AC6.1: Layer 1 命中的命令不会因为 Layer 3 有 allow 而被放行
- AC6.2: 决策链上一旦有层做出判定，后续层不执行

### 对应 F7（拒绝不终止 Loop）
- AC7.1: 权限被拒后，Agent Loop 继续运行，模型收到错误反馈
