# 上下文压缩 Spec

## 背景

Codia 的消息循环（`AgentLoop.run`）将完整对话历史全量传给 LLM API，未做大小感知和压缩处理。随着对话轮次累积，工具结果（`grep`、`read file`、`run command` 的输出）持续膨胀，最终必然超过模型上下文窗口上限，导致 API 调用失败或对话不可继续。

目标窗口为 200K token，当前无任何 token 估算或截断机制。

## 目标

- 给 Codia 装上两层上下文压缩能力，使其能在有限 token 预算中长时间工作
- 第一层（轻量预防）：超大工具结果存盘、对话中仅保留预览和文件路径
- 第二层（重量兜底）：对话接近窗口上限时调 LLM 生成结构化摘要，老旧消息压缩为摘要、近期待保留原文
- 压缩动作对用户透明但可感知（终端有提示），用户也可手动触发
- 摘要连续失败后自动熔断，避免死循环

## 功能需求

### 第一层：轻量预防

- **F1: 单结果截断** — 工具执行后，若单个结果的 `content` 字符数超过 50K，将完整结果写入磁盘文件，对话中替换为：前 500 字符预览 + 文件路径 + token 估算数。原始结果不进入消息历史。

- **F2: 单消息合计截断** — 同一轮多个工具结果合并为一条 user 消息前，若合并后的总字符数超过 200K 字符，按结果从大到小排序，依次将较大的存盘并替换为预览，直至合计降到 200K 字符以下。

- **F3: 用户消息原样保留** — 用户（`role: "user"`）发送的原始消息不参与任何压缩或截断。压缩只针对工具结果（`toolResult` / `toolResults` 中的 `content`）。

### 第二层：重量兜底

- **F4: 自动触发** — 每次 API 请求前，估算当前消息历史的 token 总量。若估算值 ≥ 187K（200K - 13K 安全余量），自动启动重量压缩。

- **F5: 摘要生成** — 调 LLM（不带工具）将较早的消息压缩为结构化摘要，近期消息保留原文。保留窗口从尾部往回数约 1 万 token 或至少保留 5 条消息（取较大者）。

- **F6: 摘要结构** — 摘要按以下固定部分组织：（1）任务目标与当前进度 （2）已做的关键决策及原因 （3）已修改的文件及改动摘要 （4）待解决的问题或待验证的假设 （5）关键发现/注意事项。

- **F7: 摘要 Prompt 约束** — 摘要请求的 system prompt 必须：（a）明确禁止调用任何工具 （b）要求先输出 `<draft>` 分析草稿，再输出 `<summary>` 正式摘要，草稿在解析时丢弃。

- **F8: 边界消息** — 压缩完成后，在摘要消息之后插入一条 `role: "user"` 的边界提示，内容明确告知模型：摘要仅作背景参考，如需文件细节请重新读取对应文件，不要依据摘要内容脑补代码。

- **F9: 手动触发** — 用户可通过 `/compress` 命令手动触发重量压缩。手动触发时不检查 token 阈值，总是执行压缩；仅保留余量（用于确定保留窗口大小）从 13K 收窄到 3K。

- **F10: 熔断机制** — 摘要 LLM 调用失败（网络错误、API 错误、摘要内容为空等）累计 3 次后，本次会话不再尝试压缩，记录告警日志并继续运行。

### Token 估算

- **F11: 近似估算** — 不引入精确 tokenizer。锚定最近一次 API 返回的 `usage.inputTokens` 作为已知基准，后续新增的消息按 `字符数 ÷ 4` 估算增量 token。首次请求无锚点时，全量按 `总字符数 ÷ 4` 估算。

## 非功能需求

- **N1: 对主循环透明** — 压缩逻辑以独立模块形式嵌入，不改动 `AgentLoop` 和 `provider` 的核心流程。压缩后的 messages 对下游调用方是无感替换。

- **N2: 持久化不丢数据** — 被存盘的工具结果和摘要写入 `~/.codia/context/` 目录，JSON 格式，按会话和时间戳组织，会话结束后可追溯。

- **N3: 压缩延迟可接受** — 轻量预防为纯字符串操作，不增加可感知延迟。重量摘要为一次 LLM 调用（无工具），延迟等同于一次普通 API round-trip。

- **N4: 降级而非崩溃** — 压缩失败不阻断主流程。轻量预防失败（如磁盘写不进去）时保留原始消息继续。重量摘要失败时计数器 +1，达到熔断阈值后跳过后续压缩。

- **N5: 可观测** — 压缩事件需通过 AgentEvent yield 给上层（TUI），用户可在终端看到压缩提示。同时记录到日志。

## 不做的事

- 不引入精确 tokenizer（如 tiktoken），仅用字符数近似估算
- 不实现摘要策略的自动调优或模型选择——始终用当前配置的模型做摘要
- 不压缩用户原始消息——即便对话极长，用户消息始终保留原文
- 不处理 system prompt 的大小——system prompt 视为固定开销，不参与压缩
- 不做增量摘要（即不把新摘要和旧摘要再合并），每次重量压缩只产出当次摘要
- 不跨会话共享摘要——每次会话独立管理自己的上下文文件

## 验收标准

- **AC1 (F1)**: 执行一个返回 60K 字符结果的工具（如 `cat` 大文件），验证对话中只出现预览 + 路径，完整内容已写入磁盘文件
- **AC2 (F2)**: 同轮执行多个工具，合并结果超 200K 字符，验证较大的结果被存盘替换，合并后总字符数 < 200K
- **AC3 (F3)**: 用户发送的消息在任意压缩后保持原文不变
- **AC4 (F4)**: 构造接近 187K 的对话历史，下一次 API 请求前观察到重量压缩被触发
- **AC5 (F5)**: 压缩后验证：较早消息变为摘要，尾部的近期消息保持原文，保留量 ≥ 1 万 token 或 ≥ 5 条
- **AC6 (F6)**: 检查生成的摘要文件，验证包含五个固定部分（目标与进度、决策、文件改动、待解决问题、关键发现）
- **AC7 (F7)**: 检查摘要请求的 prompt，验证包含"禁止调用工具"的指令和 `<draft>`/`<summary>` 标签要求
- **AC8 (F8)**: 压缩后消息列表中包含边界消息，内容提示模型需重新读取文件而非脑补
- **AC9 (F9)**: 用户输入 `/compress`，验证重量压缩总是被触发（不检查阈值）且保留余量为 3K
- **AC10 (F10)**: 模拟 3 次连续摘要失败，验证第 4 次不再尝试且日志中有告警
- **AC11 (F11)**: 发一条消息后观察 usage 事件，验证后续 token 估算是基于上次 API 返回的 inputTokens + 增量估算
- **AC12**: 压缩事件出现在终端输出中，用户可感知
