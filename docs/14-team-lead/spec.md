# Team Lead — 主 Agent 升级 Spec

## 背景

当前 Codia 的 Agent 系统（12-agent-system）支持通过 `AgentTool` 派生子 Agent，但存在以下限制：

- **子 Agent 是临时的。** 任务完成后退出，无持久化状态，无法被再次唤醒继续工作。
- **无团队概念。** 每个子 Agent 独立运行，Agent 之间没有协作通道，所有信息必须由主 Agent（Lead）中转。
- **无共享任务系统。** 子 Agent 之间没有共享的任务列表，无法看到彼此的工作进度和依赖关系。
- **Lead 没有专项模式。** 主 Agent 同时负责决策、分配、执行和代码修改，角色混杂，无法专注指挥和决策。

## 目标

将主 Agent 升级为 Team Lead，具备以下核心能力：

- 创建和管理长期存在的「小组」，成员可持久化、可唤醒，状态落盘
- 将用户目标分解为带依赖的任务，写入小组共享任务列表
- 派生多个成员并行干活，成员之间通过共享任务列表和邮箱直接协作
- 支持 coordinator 模式：Lead 专注派人和决策，代码修改完全交给成员
- 全部任务完成后，用 git 合并各成员的工作目录，冲突可解决就解决，搞不定就回滚上报

## 功能需求

### F1: 小组管理

系统提供一个「小组」对象，包含：
- 小组名称、Lead 标识、成员花名册
- 每个成员记录：角色（Lead/Worker）、工作目录路径、运行时后端类型、是否需要审批
- 小组配置持久化到 `~/.codia/teams/<team-name>/` 目录

Lead 可以创建小组、添加/移除成员、修改成员属性。小组一经创建，其数据在进程重启后仍可读取。

### F2: 成员运行时后端

成员支持两种运行时后端：
- **TMux 窗格隔离：** 在独立 tmux 窗格中运行完整 Codia 实例，提供强隔离
- **同进程轻量模式：** 在当前进程中以协程方式运行，轻量但共享地址空间

系统按环境优先级自动选择后端：tmux 可用时优先用它，不可用时回退到轻量模式。回退发生时必须以显式信息通知 Lead，不能静默降级。

### F3: 共享任务板

每个小组持有一块共享任务板，持久化为小组目录下的任务文件。所有成员可见可操作。

任务条目包含：标题、描述、状态（pending/in_progress/completed/failed）、负责人、依赖任务 ID 列表。所有成员都可以查询和更新任务，Lead 负责创建和删除任务。

### F4: 点对点消息系统

成员之间可以通过名称发送点对点消息。消息系统分为两段：
1. **名称注册表：** 小组目录下维护成员名称到邮箱文件的映射
2. **邮箱文件：** 每条消息落盘为一个条目，包含发件人、正文、时间戳（系统自动补）、是否已读（默认未读）、摘要

并发安全：邮箱文件通过锁文件（`.lock`）保护，获取锁失败时重试，锁超过一定时间视为过期自动清理。

TMux 后端额外能力：收到新消息时，如果目标成员在 tmux 窗格中运行，系统唤醒该窗格（通过 `tmux select-window`）。

### F5: 结构化协议消息

在普通文本消息基础上，系统支持几种结构化协议消息，通过 JSON 格式的 `type` 字段区分：
- **广播消息：** 发给小组所有成员
- **审批请求/响应：** 需要审批的成员将执行计划发送给 Lead，Lead 回复包含 `{"type": "approval_response", "action": "approved"|"rejected", "plan_id": "...", "reason": "..."}` 的结构，系统根据 action 字段决定放行或阻止
- **任务指派：** Lead 向成员分配任务时附带结构化任务信息

### F6: Lead 任务编排

Lead 将用户目标分解为带依赖的任务，写入共享任务板。然后根据任务数量派生相应成员，每个成员绑定到自己的 Worktree 工作目录。

全部任务完成后（所有任务状态为 completed 或 failed），Lead 负责用 git 合并各成员的工作目录：
- 冲突能自动解决的就合并
- 冲突无法解决的，回滚该成员的改动并上报
- 合并完成后汇总结果

### F7: 成员生命周期

成员完成任务自然停下后：
1. 自动将自身标记为空闲状态
2. 通过消息系统通知 Lead（消息类型为 `member_idle`）
3. 成员的工作目录和元数据保留在磁盘上（status、workDir、sessionId 等）

「上下文恢复」指成员的工作目录和配置在磁盘上保留，后续 Lead 通过消息向该成员发送新任务时，成员不需要重新创建 worktree 环境即可继续工作。任务间的具体代码更改上下文由 worktree 的 git 状态自然保留，不额外持久化 Agent 对话历史。

### F8: Coordinator 模式

Coordinator 模式通过两把锁控制开启：
1. **配置能力开关：** 在 Codia 配置中显式开启 coordinator 能力
2. **环境变量：** 用户通过环境变量（如 `CODIA_COORDINATOR=1`）在会话中主动启用

两把锁都打开才生效。开启后，发起方（Lead）的工具集被限制为：
- **保留：** 读类工具（Read、WebFetch、WebSearch、codegraph_*）、shell 工具（Bash，用于 git 合并等）、Agent 工具（派人）、Task* 工具（任务管理）、SendMessage（发消息）
- **剥夺：** Write、Edit、NotebookEdit 等所有写文件工具
- **新增（如有必要）：** 终止成员工具、合并代码工具

### F9: 团队持久化与恢复

小组及其所有子结构（成员元数据、任务板、邮箱、消息）必须能在进程退出后完整保留。重启后 Lead 可以重新加载已有小组，恢复成员状态，继续未完成的任务。

## 非功能需求

- **N1: 并发安全** — 邮箱文件写入通过锁文件（`.lock`）保证并发安全。获取锁失败时有限次重试，锁超过 30 秒视为过期自动清理。
- **N2: 无静默降级** — 后端选择发生降级时必须通过显式消息通知 Lead，包含降级原因。
- **N3: 工具隔离** — 小组协作工具（任务板、消息）仅对小组内成员可见。主入口（用户直接交互的会话）和普通子 Agent（非小组成员）看不到这些工具。
- **N4: 持久化可靠性** — 小组状态文件使用原子写入（先写临时文件再 rename），防止写入中途崩溃导致文件损坏。

## 不做的事

- 跨机器的分布式团队（本步仅限单机）
- 成员间实时流式通信（消息为异步落盘模式，不保证实时送达）
- 复杂任务依赖约束（仅支持简单的「依赖任务 ID 列表」，不实现 DAG 校验、循环检测等）

## 验收标准

- **AC1（F1-小组管理）：** 执行创建小组操作后，`~/.codia/teams/<name>/` 目录存在且包含 `group.json` 和 `members/` 子目录
- **AC2（F2-后端选择）：** 在 tmux 环境下创建成员，成员运行在独立窗格中；在没有 tmux 的环境下创建成员，Lead 收到包含降级原因的通知消息
- **AC3（F3-共享任务）：** 成员 A 创建任务后，成员 B 可以通过任务工具查询到该任务并修改其状态
- **AC4（F4-消息系统）：** 成员 A 向成员 B 发消息后，B 的邮箱文件中出现一条已格式化的消息记录（含时间戳、默认未读）
- **AC5（F5-审批流程）：** 需要审批的成员提交计划后，Lead 回复 `action: "approved"` 结构，成员继续执行；回复 `action: "rejected"`，成员停止执行
- **AC6（F6-任务编排）：** Lead 创建 2 个依赖任务、派生 2 个成员分别完成、全部完成后执行 git 合并，两个成员的工作结果都出现在合并后的分支中
- **AC7（F7-成员恢复）：** 成员完成任务标记空闲后，Lead 再发消息指派新任务，成员从磁盘恢复上下文继续工作（不需重新创建 worktree 环境）
- **AC8（F8-Coordinator 模式）：** 开启 coordinator 模式后，Lead 调用 Write 工具返回「工具不可用」；关闭后恢复正常
- **AC9（F9-持久化恢复）：** 小组创建并写入数据后重启进程，重新加载小组，任务板、成员花名册和消息历史仍完整可读
