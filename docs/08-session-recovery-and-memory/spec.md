# 会话恢复与分层记忆管理 Spec

## 背景

当前 Codia 已具备基础的会话持久化、System Prompt 拼装和上下文压缩能力，但新会话启动时仍主要依赖当前目录和即时输入，缺少对“项目长期知识”和“用户稳定偏好”的自动恢复。结果是 Agent 在跨会话工作时容易重复询问、重复探索，遇到中断后也难以平滑续接之前的工作上下文。

本次迭代要补上三层恢复机制：
1. **项目指令文件**：提供可手写、可分层覆盖的长期规则与约束
2. **会话存档**：提供最近工作记忆与可恢复的历史上下文
3. **自动笔记**：沉淀用户偏好、纠正反馈、项目知识和参考资料，作为长期记忆来源

三者共同组成新会话启动时的恢复管线，使 Agent 在进入首轮推理前，已经拿到与当前项目和当前用户相关的必要背景。

## 目标

- 在新会话启动时，自动恢复与当前项目相关的长期规则和已沉淀的关键记忆；在恢复旧会话时，自动恢复可继续使用的历史上下文
- 让 Agent 在跨会话使用时减少重复询问、重复搜索和重复确认
- 将“手写指令”“工作会话”“自动沉淀记忆”分层管理，避免职责混淆
- 在恢复链路中提供可降级的容错能力，使局部损坏或局部缺失不会阻止会话启动
- 为后续扩展更强的记忆能力保留空间，但本阶段只实现本地文件驱动的恢复方案

## 功能需求

- F1: 系统应在会话启动前自动加载三层项目指令文件：`<project-root>/MEWCODE.md`、`<project-root>/.mewcode/MEWCODE.md`、`~/.mewcode/MEWCODE.md`，并按预定义优先级拼接后注入上下文，使高优先级规则先于低优先级规则出现。
- F2: 项目指令文件应支持通过 `@include` 引用其他指令文件，并在展开过程中处理嵌套深度限制、重复访问检测和越界路径拦截，避免无限递归或读取未授权范围外文件。
- F3: 系统应在 `~/.codia/projects/<project-id>/sessions/` 目录中，以追加写方式持续保存会话消息，使正常对话过程中的存档开销低、崩溃时最多只丢失最后一条未完整写入的数据。
- F4: 系统应基于会话存档本身计算会话标识、标题预览、消息数量和最近活动时间，而不是依赖独立的同步元数据文件。
- F5: 系统在恢复历史会话时，应兼容当前支持的单工具结果格式和同轮多工具结果格式；在遇到损坏记录或尾部未闭合的工具调用相关记录时，应跳过或截断到最后一个可安全恢复的位置，而不是让整个会话恢复失败。
- F6: 系统在把历史会话重新送入模型前，应先处理长时间中断和上下文超长两类情况：当会话距上次活动已超过预设阈值时，插入一条明确的时间跨度提醒；当恢复上下文超出安全范围时，应只执行一次压缩并避免与正常请求链路重复压缩。
- F7: 系统应定期清理超过 30 天保留期的旧会话存档，防止本地会话目录无限增长。
- F8: 系统应在每轮 Agent Loop 自然结束后，异步分析本轮对话是否产生可复用知识，并按预定义类别写入自动笔记目录。
- F9: 自动笔记至少应支持用户偏好、纠正反馈、项目知识、参考资料四类，并保留足够的结构信息，使后续可以区分来源和用途。
- F10: 系统应避免为同一事实反复生成重复笔记；当新信息与既有笔记语义重叠时，应由 LLM 在新增、更新或跳过之间做出去重决策，而不是无上限新增。
- F11: 系统应在处理用户请求前，分别加载与当前项目相关的项目级记忆索引和用户级记忆索引，并将其作为会话前置上下文注入；项目级记忆存放在 `~/.codia/projects/<project-id>/memory/`，用户级记忆存放在 `~/.codia/memory/`，注入时遵循固定顺序。
- F12: 记忆索引文件应限制在 200 行 / 25KB 以内，避免长期积累后反过来挤占主任务上下文窗口。
- F13: 当项目指令、会话存档或自动笔记中的某一类数据缺失、损坏或暂不可用时，系统仍应以剩余可用信息继续启动，而不是整体中断。

## 非功能需求

- N1: 新会话恢复流程应保持可接受的启动开销；在本地文件数量正常增长的前提下，不应因为加载指令、扫描索引或恢复最近会话而让启动体验明显卡顿。
- N2: 本阶段所有恢复能力都应基于本地文件系统实现，不引入向量数据库、外部检索服务或远程同步依赖。
- N3: 恢复链路的关键动作应具备可追踪性，使开发者能够定位“加载了哪些指令”“恢复了哪个会话”“注入了哪些记忆来源”等结果。
- N4: 单条坏记录、单个坏文件或单个异常引用，不应污染整个恢复流程；系统应尽量把失败范围限制在出错对象本身。
- N5: 项目指令引用与项目级记忆读取必须受当前项目边界约束，不能因为路径引用而读取项目目录之外的任意文件；用户级入口仅允许读取 `~/.mewcode/` 子树中的文件。
- N6: 注入模型前使用的记忆索引文件应有明确体积上限，控制在 200 行 / 25KB 以内，避免长期积累后反过来挤占主任务上下文窗口。
- N7: 自动笔记分类在不同轮次之间应保持稳定含义，避免同一类事实在不同时间被归入不同类别，导致后续注入和维护混乱。
- N8: 本阶段设计应允许未来接入更强的检索、共享或同步能力，但当前实现不应预先引入这些能力所需的复杂基础设施。

## 不做的事

- 不引入向量数据库、语义检索服务或 RAG 流程
- 不做多用户/团队级记忆共享、同步或权限管理
- 不做跨设备或云端会话同步
- 不把项目指令、会话存档和自动笔记统一抽象成单一记忆模型
- 不在本阶段解决所有历史会话压缩策略，只要求恢复路径与正常请求路径协同，确保超限历史最多只被压缩一次
- 不要求自动笔记具备完全自动的真伪校验能力，去重与更新以“避免明显重复、可维护”为目标
- 不要求恢复系统替代用户显式提供上下文；当关键信息本就不存在时，Agent 仍可以继续提问
- 不扩展到与当前项目无关的全局知识管理能力，除用户级记忆外，其余恢复均以当前项目上下文为边界

## 验收标准

- AC1: 当 `<project-root>/MEWCODE.md`、`<project-root>/.mewcode/MEWCODE.md` 和 `~/.mewcode/MEWCODE.md` 都存在时，新会话启动后注入的指令内容应按既定优先级排序；当某个指令文件包含合法 `@include` 时，被引用内容应正确展开；当引用形成环路、超过嵌套深度或试图越出允许范围时，应拦截异常引用而不导致整体加载失败。
- AC2: 当系统持续进行多轮对话时，会话消息应以追加写方式保存在 `~/.codia/projects/<project-id>/sessions/` 下的单个 JSONL 文件中；在不依赖独立元数据文件的前提下，系统仍能从存档文件本身恢复会话标识、预览、消息数量和最近活动时间。
- AC3: 当会话存档中同时包含旧格式单工具结果记录与新格式同轮多工具结果记录时，系统恢复会话时应正确兼容两种格式；若文件中包含损坏行、半写入尾行或尾部未闭合的工具调用相关记录，应跳过或截断不可恢复部分，并成功恢复此前的完整历史。
- AC4: 当待恢复会话距上次活动已超过预设阈值时，恢复后的上下文中应包含一条明确提示时间跨度的提醒信息；当待恢复上下文超过安全范围时，系统在进入首轮模型请求前应只执行一次压缩，不得与正常请求链路重复压缩。
- AC5: 当本地存在超过 30 天保留期的旧会话存档时，系统应在预定清理时机删除这些过期文件，且不影响保留期内会话的正常列出与恢复。
- AC6: 当一轮 Agent Loop 自然结束且本轮对话产生可复用知识时，系统应异步生成或更新自动笔记，并将其归入用户偏好、纠正反馈、项目知识、参考资料之一；当新旧事实明显重复时，应由 LLM 决定更新、复用或跳过，而不是无界增长。
- AC7: 当用户级与项目级记忆索引存在时，系统在处理用户首条请求前应已按固定顺序将两类索引注入上下文；项目级索引来自 `~/.codia/projects/<project-id>/memory/`，用户级索引来自 `~/.codia/memory/`，不要求运行时临时扫描所有原始笔记全文。
- AC8: 当记忆索引增长时，系统应将索引控制在 200 行 / 25KB 以内，同时保留可用的最新摘要。
- AC9: 当项目指令、会话存档和自动笔记中的任意一类数据缺失、部分损坏或暂不可用时，系统仍能以剩余可用恢复信息完成新会话启动，而不是因为单点异常完全不可用。
