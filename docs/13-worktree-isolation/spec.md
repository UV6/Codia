# Worktree 隔离 Spec

## 背景

Codia 的 Agent 系统已支持子 Agent 委派（定义式/Fork 式），但所有子 Agent 共享同一工作目录。当主 Agent 和子 Agent 并行操作时，会出现文件写入冲突：两个进程可能同时读取、修改、写入同一个文件，导致数据丢失或状态不一致。

Git worktree 机制允许从同一仓库创建多个独立工作目录，每个目录有自己的文件系统视图和分支，共享同一个 `.git` 对象库。这天然适合解决 Agent 并行隔离问题。

## 目标

- 每个子 Agent 在独立的工作目录中操作，互不干扰
- 子 Agent 声明隔离需求后，创建、初始化、切换、清理全自动，零心智负担
- 目录名来自 LLM 输入，严格校验防路径遍历
- 退出时自动检测未提交修改，防止误删导致工作丢失
- 与现有架构无缝对接，复用已有的 `cwd` 参数链路，不改动核心循环逻辑

## 功能需求

### F1: 目录名校验

接收目录名输入，执行安全校验：

- **允许字符集**：`[a-zA-Z0-9._-]` 加 `/`（嵌套分隔），无空格
- **长度限制**：全路径 ≤ 64 字符
- **独立段检查**：`.` 和 `..` 作为独立路径段拒绝（`./foo`、`foo/../bar` 非法）
- **语法检查**：拒绝以 `/` 开头（绝对路径）、连续 `//`（空段）、以 `/` 结尾
- **校验失败**：返回明确错误信息，不触碰文件系统

### F2: 工作目录创建

在校验通过的目录名下创建 git worktree：

- **目录位置**：`<repoRoot>/.codia/worktrees/<name>/`
- **分支命名**：`worktree-{flatSlug}`，其中 `flatSlug` 将 name 中的 `/` 替换为 `+`
- **baseBranch**：作为参数显式传入（如 `main`），不写死 HEAD
- **创建命令**：`git worktree add -B <branch> <path> <baseBranch>`（`-B`：不存在则创建，存在则强制重置，防孤儿分支残留导致 `-b` 报错）
- **目录已存在**：只读文件系统确认可用性，不重复调 git

### F3: 环境初始化

工作目录创建后执行：

- **配置文件复制**：将项目根目录的运行时配置（如 `.claude/` 下的 settings、rules、memory、CLAUDE.md 等）复制到工作目录
- **Git hooks**：检查 `git config core.hooksPath`，存在则复制其指向的目录内容；不存在则复制 `<gitdir>/hooks/`。优先 `.husky/`
- **依赖软链**：对 `node_modules/` 等大型依赖目录建立软链接，避免重复安装
- **忽略文件补全**：按显式规则列表补上 `.gitignore` 忽略但需要的文件（此特性预留，初始实现不包含具体规则，仅保留扩展点）

### F4: 工作目录进入

- 不修改进程 `cwd`（不调 `chdir`）
- 返回工作目录的绝对路径
- 上层将所有工具调用的 `cwd` 参数指向该路径
- 所有路径相关缓存（文件内容、系统提示词、项目指令、记忆）使用绝对路径作为 key

### F5: 工作目录退出与删除

退出时变更检查：

- **保护范围**：检查"未提交修改"和"本地比创建时 head commit 多的新 commit"——commit 了即视为有价值工作
- **默认拒绝删除**：有变更时拒绝，返回保护信息（变更文件列表、新增 commit 数）
- **不存在处理**：操作不存在的 worktree 名称时，返回 `WorktreeNotFoundError`，不执行任何文件系统操作
- **强制删除**：提供 force 参数跳过保护，执行 `git worktree remove --force` 并删除对应分支
- **保留模式**：仅标记状态，目录留磁盘供后续恢复

### F6: 子 Agent 隔离集成

- **Frontmatter 扩展**：`AgentRoleFrontmatter` 新增 `isolation?: "worktree"` 字段
- **自动创建**：子 Agent 启动时，若角色声明 `isolation: worktree`，自动创建 worktree 并将路径注入 `SubAgentConfig.cwd`
- **上下文通知**：创建 worktree 后，向前置任务描述注入通知文本：

  > 你在一个隔离的 Git Worktree 中工作。工作目录：`<worktree-path>`。
  > - 父 Agent 传来的文件路径指向主工作目录，请翻译为你本地的对应路径
  > - 在编辑任何文件之前，必须重新读取该文件以确保你看到的是最新内容
  > - 完成后，你的所有改动都在此隔离目录中，父 Agent 会决定是否合并

- **完成清理**：子 Agent 执行完毕，根据变更情况自动决定保留或清理
- **向后兼容**：未声明 `isolation` 的角色行为完全不变

### F7: 后台过期清理

- **命名模式区分**：`agent-a*`（Agent 自动创建）和 `wf_*`（Workflow 自动创建）模式的目录可被清理；用户手动创建的目录（非自动模式命名）永不清理
- **三层过滤顺序**：① 命名模式匹配 → ② 过期检查（最后修改时间 vs 可配置的 `cutoffDate`） → ③ 变更保护（有未提交修改则跳过）
- **可配置过期阈值**：`cutoffDate` 作为参数传入，不硬编码
- **清理操作**：`git worktree remove` + 删除分支，记录清理日志

## 非功能需求

### N1: 安全性
- 所有文件系统操作必须基于校验后的路径，不允许任何情况下拼接未校验的用户输入作为路径
- 路径遍历防护覆盖创建、进入、删除三个入口

### N2: 可靠性
- 创建 worktree 失败时不得遗留半初始化状态（目录、分支、git 元数据任何一项失败需回滚）
- 删除失败时不得破坏 git 仓库的完整性（不能出现分支已删但目录残存，或反之）

### N3: 可观测性
- 每个生命周期操作（创建、进入、退出、删除、清理）都产生可读日志，包含操作名称、目标路径、耗时、结果
- 异常情况产生警告级别日志（如目录已存在直接复用、清理时发现未提交修改跳过）
- 日志通过项目现有日志基础设施输出，无需新建日志框架

### N4: 性能目标（设计目标，非硬性门禁，性能验证在后续阶段进行）
- 创建操作（不含 `git worktree add` 本身耗时）应在 2 秒内完成
- 目录已存在时的复用路径应在 50ms 内完成

### N5: 可测试性
- 每个模块独立可测：校验器、创建器、初始化器、清理器各自接受依赖注入
- git 操作通过接口抽象，测试可用 mock 替代真实 git 调用

### N6: 平台兼容
- 支持 macOS 和 Linux，路径分隔符统一用 `/`

## 不做的事

- Worktree 之间的合并策略（上层用 `git merge` 决定）
- 跨目录的代码同步或共享状态
- 多 Agent 并行编排调度（留给后续章节）
- 网络隔离、容器级隔离等更重的隔离方案

## 验收标准

### AC1 (对应 F1): 目录名校验
- 合法名 `agent-a3f2b1c`、`sub/agent-x` 通过校验
- `../escape`、`./foo`、`/absolute`、`a//b`、`.` 、`..` 被拒绝并返回明确错误
- 名称超过 64 字符被拒绝

### AC2 (对应 F2): 工作目录创建
- 调用创建后，`.codia/worktrees/<name>/` 目录存在且可读写
- 目录内 `git branch --show-current` 返回 `worktree-{flatSlug}`
- 重复创建同一名称不报错（幂等）
- 指定 `baseBranch` 为 `main` 时，工作目录基于 `main` 而非当前 HEAD

### AC3 (对应 F3): 环境初始化
- 工作目录中存在与主目录相同的 `.claude/` 配置副本
- `node_modules/` 以软链接形式存在
- `core.hooksPath` 指向的 hooks 已正确复制

### AC4 (对应 F4): 工作目录进入
- 进入后返回的路径是绝对路径
- 进程 `cwd` 不变（通过 `process.cwd()` 验证）

### AC5 (对应 F5): 退出与删除
- 工作目录有未提交修改时，默认删除被拒绝
- 工作目录有新增 commit（未合并回主分支）时，默认删除被拒绝
- 干净目录可正常删除
- force 模式无论有无变更都能删除

### AC6 (对应 F6): 子 Agent 集成
- 角色声明 `isolation: worktree` 后，子 Agent 创建时自动获得隔离 worktree 路径作为 cwd
- 子 Agent 收到的任务描述中包含上下文通知文本
- 未声明 isolation 的角色行为不变

### AC7 (对应 F7): 后台清理
- `agent-a*` 和 `wf_*` 模式的过期目录被清理
- 手动创建目录（非自动模式命名）即使过期也不被清理
- 有未提交修改的过期目录跳过不清理
- 指定 `cutoffDate=1h` 时，1 小时前创建的目录被清理
