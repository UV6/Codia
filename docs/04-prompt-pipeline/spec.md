# 提示词管线 Spec

## 背景

Codia 目前的系统提示是一行硬编码字符串 `"You are Codia, a helpful CLI AI assistant."`，外加 plan mode 常量拼接。随着功能增长（工具系统、Agent Loop、plan mode），模型需要更结构化的行为指引，但当前做法有几个问题：

1. **缓存失效：** 所有指令混在一起，任何一个环境变量变化（如日期、cwd）都导致整个 system prompt 变化，Anthropic 前缀缓存无法命中，每次请求都重新处理全部指令
2. **指令遵守率不稳定：** "优先用专用工具"、"编辑前必先读"这类关键规则淹没在长文本中，模型有时忽略
3. **注入信息无结构化：** plan mode 状态、工具列表等动态信息直接拼接到 system 消息中，没有统一的注入规范和频率控制
4. **扩展困难：** 加一条全局规则就要改代码，模块边界模糊

## 目标

- 构建结构化的 System Prompt，按职责拆成模块，按优先级拼装，方便后续插新模块
- 稳定模块走缓存通道（前缀命中），动态信息通过 `<system-reminder>` 消息注入，不破坏缓存
- 在工具描述和全局指令里双重强化关键规则
- 会话级开关（如 plan mode）的指令首轮完整注入，后续轮次精简为标签
- 可观测缓存命中情况，用 API 返回的缓存字段验证策略是否生效

## 功能需求

### F1: 模块化 System Prompt 构造

将全局指令按职责拆为七个固定模块，按优先级顺序拼装：

| 优先级 | 模块 | 职责 |
|--------|------|------|
| 1 | 身份 | 模型自称、项目背景、能力边界 |
| 2 | 系统约束 | 不猜测、不混淆、先问后做 |
| 3 | 任务模式 | 目标驱动、验证闭环、计划优先 |
| 4 | 动作执行 | 编辑前必读、不改无关代码、风格匹配 |
| 5 | 工具使用 | 优先用专用工具、codegraph 优先、Bash 使用规范 |
| 6 | 语气风格 | 语言选择、输出简洁度、无道歉套话 |
| 7 | 文本输出 | 格式规范、代码引用格式、禁止冗余输出 |

七个固定模块之后预留三个可选模块的插入位：自定义指令、已激活的 Skill 列表、长期记忆。本迭代不填充这三个模块的内容（见"不做的事"），仅留好架构扩展点。

模块间以空行分隔，整体从前往后按优先级拼装。

### F2: 稳定内容与变化内容的分离

- 七个固定模块的内容由代码定义，不随运行环境变化——走 Anthropic 的 `system` 参数字段，作为缓存前缀
- 环境信息（系统运行环境、Git 上下文）不作为固定模块内容，改为通过 `<system-reminder>` 消息注入。环境信息仅在首轮（round=0）注入，后续轮次不刷新——这是缓存优化的设计取舍。模型在工作目录变化时仍可从工具执行结果中感知路径信息，且工具调用时 `cwd` 始终为实时值
- 工具描述列表不拼入 system prompt，通过 API 的 `tools` 参数传递

### F3: 关键规则双重强化

对于"优先用专用工具"、"编辑前必先读"等关键约定：
- 全局指令中写入（模块 5: 工具使用、模块 4: 动作执行）
- 对应工具的 `description` 字段中也写入提醒
- 保证规则出现不少于两次，提高遵守率

### F4: `<system-reminder>` 注入机制

- 以 `user` role 消息的形式注入，内容用 `<system-reminder>` XML 标签包裹
- 注入消息排在对话历史的合适位置（最新一条之前），不混入 system prompt
- 模型看到该标签后将其识别为系统补充指令（非用户输入），不回复，纳入工作上下文
- 适用于：MCP 上下线通知、可用 Skill 列表、Agent 类型声明、环境信息、plan mode 状态标签等

## 非功能需求

### N1: 性能—缓存友好

- 固定模块构成的 system prompt 在不同请求间保持内容不变，使其可作为 Anthropic prompt cache 的前缀命中
- `<system-reminder>` 消息排在对话历史末尾（最新 user 消息之前），变化时只破坏极小范围的缓存，不影响前缀
- 缓存命中情况可通过 API 返回的 `usage.cache_creation_input_tokens` 和 `usage.cache_read_input_tokens` 字段观测

### N2: 可扩展性

- 新增一个全局指令模块只需：定义新 Section，设置优先级，追加到 Builder 的列表末尾
- 不需要修改现有模块代码或拼装逻辑
- 未来新增 MCP 接入、自动记忆、项目指令加载等功能时，只需增加新的 Section 或 `<system-reminder>` 注入源，不重构核心管线

### N3: 可观测性

- System prompt 的最终内容可通过日志或调试接口完整输出
- 缓存命中/未命中可通过 API 返回字段定量追踪
- 每次注入 `<system-reminder>` 时记录其来源和轮次，方便排查

### N4: 兼容性

- 同时支持 Anthropic API（system 参数数组 + messages 数组）和 OpenAI API（system role message + messages 数组）
- 内部 Message 类型不变（`role: "user" | "assistant" | "system"`），`<system-reminder>` 使用现有 `user` role，不新增 role 类型

## 不做的事

- 项目指令文件加载：不读取项目目录下的 `.codia/instructions.md` 或类似文件
- 自动记忆系统：不从对话中自动提取和持久化记忆
- 真实 MCP 接入：不实现 MCP 服务器的真实连接和通信，只预设 `<system-reminder>` 注入位
- 自动化缓存效果评估：不写自动对比脚本，但会提供手动观测方法和典型场景清单

## 验收标准

| 编号 | 内容 | 对应需求 |
|------|------|---------|
| AC1 | 调用 Builder 输出七个固定模块的完整 system prompt 文本，模块按优先级顺序排列 | F1 |
| AC2 | 可单独替换任意模块内容，不改其他模块 | F1 |
| AC3 | 固定模块内容不含任何运行时变量（日期、路径、git 状态、工具列表） | F2 |
| AC4 | 环境信息通过 `<system-reminder>` 消息注入，不出现在 system prompt 中 | F2 |
| AC5 | 工具列表通过 API `tools` 参数传递，不拼入 system prompt | F2 |
| AC6 | "优先用专用工具"、"编辑前必先读"等关键规则在全局指令和工具描述中均出现 | F3 |
| AC7 | 注入消息 role 为 `user`，内容以 `<system-reminder>` 开头、`</system-reminder>` 结尾 | F4 |
| AC8 | 注入消息排在对话历史末尾（最新 user 消息之前），不混入 system prompt | F4 |
| AC9 | Anthropic 和 OpenAI provider 均正确适配新的 message 结构 | N4 |
| AC10 | 缓存命中/未命中可通过日志或调试接口观测 | N3 |
| AC11 | 提供至少 3 个典型场景的手动对比清单（含预期缓存命中情况） | — |
